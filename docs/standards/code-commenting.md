# 代码注释规范

本文档约束 `cc-snowflake-id-go` 的代码注释写法，目标是让源码、GoDoc、测试和仓库文档保持一致口径。

## 适用范围

- 所有导出常量、变量、类型、函数、方法
- 少量关键内部逻辑
- 包级说明与仓库入口文档
- 复杂测试、基准测试、集成测试

## 生产代码必须满足的规则

1. 所有导出符号都必须有 GoDoc 注释。
2. 注释必须以符号名开头。
3. 注释必须说明职责、边界或约束，不能只把代码再念一遍。
4. 导出 API 变更时，必须同步更新源码注释和 `README.md`。

## 生产代码推荐写法

- 常量：说明语义、单位、默认值或边界。
- 错误变量：说明它代表的失败类别，而不是只翻译字符串。
- 类型：说明它管理什么状态、适合什么场景、是否涉及并发或生命周期。
- 函数：说明输入约束、输出含义和失败条件。
- 方法：优先说明状态转换、副作用和调用时机。

## 内部注释什么时候允许写

只有在下面这些场景才写内部注释：

- 并发控制顺序不直观
- 时钟、围栏、租约等时序约束容易误判
- SQL 或状态机逻辑存在关键不变量
- 需要解释为什么这样做，而不是代码本身做了什么

## 测试代码注释规则

测试代码不是“不许写注释”，而是“不强制像导出 API 那样写 GoDoc”。

默认规则：

- 简单单元测试如果测试名已经清楚表达行为，可以不写注释。
- 复杂测试必须补注释，尤其是并发、取消、续租、fence、时钟漂移、失败恢复和集成测试。
- 注释优先说明场景意图、关键约束和为什么这样断言。
- 不要求给每个 `TestXxx`、`BenchmarkXxx` 都补函数头注释。

测试代码推荐写法：

- 在复杂测试前用 1 到 2 行注释说明场景。
- 对关键的 arrange 阶段补注释，说明 fake/mock 在模拟什么时序条件。
- 对“必须先发生 A，才能允许 B”的断言补注释，明确顺序约束。
- 对时间窗口、魔法数字和 build tag 说明其测试意义。

测试代码不允许的写法：

- “调用 Next 生成 ID”
- “这里断言结果”
- “进入 if 分支”
- 和测试函数名重复、但没有新增信息的注释

## 评审要求

- 新增导出 API 没有注释，不能合并。
- 修改了发号、租约、telemetry 行为，但注释没更新，不能合并。
- 新增复杂测试但没有说明场景意图、关键时序或外部依赖，应该补注释后再合并。
- 文档与源码描述冲突时，修正文档和注释属于当前变更的一部分，不能留到后补。
